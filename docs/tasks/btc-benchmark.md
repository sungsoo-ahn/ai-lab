# BTC Benchmark

Date: 2026-06-10
Status: active

## Summary

BTC Benchmark is the frozen BTC 1h task. It is separate from any scientist scheme that attempts it.

The task exists to exercise the AI Lab platform on a concrete benchmark with an automated referee. The platform question is not only "what score did we get?" It is also "can two different scientist schemes attack the same benchmark under a comparable contract?"

## Candidate Metric

The main metric is the benchmark referee's development score. Scores are valid only when produced without changing referee semantics.

## Constraints

- Do not modify referee scoring, cost model, causality checks, or holdout policy.
- External submissions require explicit user approval.

## Task Contract

| Area | Contract |
| --- | --- |
| Dataset family | BTC hourly benchmark data from the configured source checkout. |
| Primary evaluator | Benchmark referee development score. |
| Valid outputs | Reports and leaderboards generated by the approved benchmark/referee path. |
| Invalid shortcuts | Changing scoring semantics, leaking future data, weakening costs, changing holdout policy, or submitting externally without approval. |
| Comparison unit | Evaluation cell: one scientist scheme plus declared skills, taste, hypotheses, target metric, constraints, and run evidence. |

## Why This Task Is Useful For Platform Work

BTC Benchmark is narrow enough to run repeatedly but rich enough to expose platform issues:

- a scientist can overfit a public development score;
- causal feature discipline matters;
- transaction costs and turnover can reverse apparent improvements;
- simple baselines are hard to beat robustly;
- run artifacts are small enough to inspect;
- fixed command lines can make unattended work practical.

This makes it a good first task for testing whether the lab can run comparable cells without human micromanagement.

## Current Baseline Expectations

Every BTC cell should preserve:

- source status and benchmark readiness checks;
- the exact command or run spec;
- preflight baseline output;
- candidate reports and leaderboards when experiments run;
- synthesis notes explaining accepted, rejected, and unresolved hypotheses.

If a cell lacks these artifacts, treat its result as incomplete even if a command returned successfully.

## Latest Evidence

The latest AutoResearch overnight run, `btc-overnight-20260610T151519Z-autoresearch`, completed on 2026-06-10 UTC. It confirmed `candidate_always_long_default_gates` at net `1.5005`, Sharpe `0.770`, max drawdown `-0.6670`, and turnover `1.0`.

Interpret this as the BTC development-score floor rather than a novel strategy: it nearly matches the benchmark buy-and-hold reference after one opening cost and still has severe drawdown and funding drag. The follow-up proposal is to rank Cycle 2 candidates against buy-and-hold-relative evidence such as drawdown reduction, funding-aware net, cost stress, next-open execution, per-year concentration, and random percentile.
